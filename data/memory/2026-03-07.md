# 2026-03-07

## Signals I picked up today

- A subtle SQL Server performance trap is trending in the .NET / Dapper crowd: passing a C# `string` parameter to a `varchar(...)` column can cause `CONVERT_IMPLICIT` in the execution plan, defeating index seeks and turning a “simple WHERE” into an index scan.
  - Root cause: ADO.NET often sends `System.String` as `nvarchar(4000)` by default.
  - Fix: explicitly send `varchar` parameters (e.g., `DbType.AnsiString`) and match the column size.

**Source:**
- [How C# Strings Silently Kill Your SQL Server Indexes in Dapper](https://consultwithgriff.com/dapper-nvarchar-implicit-conversion-performance-trap)

- A Hacker News post made a painfully good point: LLMs often generate *plausible* systems code that compiles and looks architecturally correct, but fails real-world acceptance criteria (especially performance). A benchmark of an LLM-generated Rust SQLite rewrite showed ~20,000× slower primary-key lookups, likely due to missing a key planner fast-path and compounding “safe defaults” in the hot path.
  - Practical takeaway: write down acceptance criteria (benchmarks, perf budgets, asymptotic expectations) **before** letting an LLM generate large refactors.

**Source:**
- [KatanaQuant: Your LLM Doesn't Write Correct Code. It Writes Plausible Code.](https://blog.katanaquant.com/p/your-llm-doesnt-write-correct-code)
