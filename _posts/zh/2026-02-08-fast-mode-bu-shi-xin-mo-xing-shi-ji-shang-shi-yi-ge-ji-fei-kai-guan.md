我很喜歡任何能讓迭代變快的東西。

但我也很討厭團隊把「速度選項」當成魔法。

Claude Code 的 fast mode 就是一個典型案例：很多人看到它、打開它，然後開始用「換了一個模型」的方式在聊。

不是。

它更接近：把 **延遲的優先級** 跟 **計費的優先級** 用一個按鈕包起來。

這聽起來很龜毛，但你只要遇過下面任何一種情境，就會知道這種細節會咬人：
- 成本突然飆高，你要解釋原因
- 團隊要做 live debugging，需要回應夠快
- 或者有人在一個很長的 session 中途切換，結果被收了「整段對話重新算一次」的高價

## fast mode 到底是什麼

按照文件說法，fast mode 使用的是**同一個底層模型**，只是 API 端做了不同的配置，用「速度」換「成本效率」。

所以你應該用不同方式操作它：

- 你要的是 **快速回饋迴圈**（快速改 code、反覆試、現場除錯）→ fast mode 很合理。
- 你要的是 **長時間自動跑**（batch、偏 CI 的工作）→ fast mode 通常不是好預設。

一句話：fast mode 不是「更聰明」，而是「更快，但你要付出代價」。

## 真正的地雷：在長對話中途切換

我反而不是先看價格表。

我最在意的是文件提到的那句：**在對話中途切到 fast mode，會以 uncached input 的方式計價整段對話的 context**。

翻成人話就是：

> 你在一個已經聊很久的 session 裡才打開 fast mode，你可能會用更貴的單價，把整段歷史訊息再算一次。

這種事情一定會出現在成本檢討會上。

所以我會在團隊 wiki 寫一條很無聊但很實用的規則：

- 需要 fast mode → **一開始就開**。
- 在乎花費 → **不要把它當主題切換器** 隨手開關。

## 我怎麼看 agent 系統裡的「速度旋鈕」

這不只是某個 CLI feature。

在 agent 系統裡，你通常至少有三個互相獨立的槓桿：

1) **模型選擇**（能力 / 成本曲線）
2) **effort / reasoning budget**（願意讓它想多久）
3) **延遲配置**（供應商端對速度的優先級）

很多團隊會把這三個混在一起，然後抱怨系統「忽快忽慢、忽貴忽便宜、結果還不穩」。

我比較希望把模式講清楚：

- 「互動模式」：貴一點沒關係，但回應要快，人一直在 loop 裡。
- 「批次模式」：便宜一點，可以想久一點，放背景跑。

而且重點是：把 mode 寫進 log。

因為六週後有人問你「為什麼這次 run 成本 4 倍？」你要給的是可驗證的答案，不是氣氛。

## 我的立場

我支持 fast mode。

我只是反對「不小心就 fast mode」。

如果你在建 agent 工作流，把速度當成 operating decision 會舒服很多：
- 有意識地設定
- 一個 session 裡盡量保持穩定
- 像 SLO tradeoff 一樣去衡量
