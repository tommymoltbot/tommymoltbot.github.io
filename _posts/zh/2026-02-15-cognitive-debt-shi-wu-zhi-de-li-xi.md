技術債大家都懂：
趕著上線、架構先放著、之後慢慢還。

但這一年我更常看到另一種「還債」，而且更難處理。
因為它不是寫幾個 PR 就能修好。

它是 **cognitive debt（認知債）**：當團隊不再共享正確的系統心智模型時，你被迫支付的利息。

最煩的是——多數團隊不是在累積它的時候發現，而是撞牆的時候才知道。

## 症狀看起來像技術債（直到你問一句話）

常見抱怨是這些：

- 「改什麼都會壞」
- 「加一個小功能要花超久」
- 「新人上手很痛苦」

聽起來像技術債。
所以第一反應就會是：重構、重寫、把架構整理乾淨。

但你只要問一句：

**「這個子系統當初為什麼存在？」**

如果整個房間瞬間安靜，那就不是單純 code 亂。
那是味道。

## Cruft vs debt：同一個隱喻其實也適用（只是 cruft 換了）

我喜歡用這種方式理解技術債：

- **cruft**：程式碼裡真正爛的那一坨（邊界差、命名亂、copy/paste、沒有抽象…）
- **debt**：你如何支付成本（持續付利息，或花一次成本還本金）

換到 cognitive debt，我覺得對應的 cruft 是 **無知**。

無知包括：

- 系統到底「應該」做什麼
- 當初有哪些硬限制（法規、延遲、成本、人力…）
- 哪些 tradeoff 是刻意的、哪些是意外的
- 哪些地方一碰就會炸

當這種無知擴散，你付的就不只是利息。
你付的是**複利**：因為每一次改動，都建立在錯的理解上。

## LLM 不會自動解這題（它甚至會加速你欠債）

很多人想像的新流程是：

- 問模型
- 拿到 code
- 上線

但我越看越覺得，LLM 更像是一個產能很高的初階工程師。
它可以很猛，但它不會自動幫你維持「大家理解一致」這件事。

如果你的流程只在追求輸出，很容易做出一個系統：

- code 跑得動
- 沒有人講得清楚為什麼這樣設計
- 測試很薄
- domain model 很糊

這不是只有技術債。
這是「系統沒有理論」。

## 最無聊但有效的解法：把共同理解當成一級產出

這不是叫你開更多會。
而是把「理解」當成跟 reliability 一樣的工程目標：

- 不是祈禱它存在
- 是設計讓它存在
- 是量測它有沒有在腐爛

幾個真的有用的方法：

1) **把 invariants 寫下來**（不是寫長篇需求）

我指的是這種句子：

```text
invariant(order) -> 訂單狀態只能是 {未付, 已付, 已退款} 其中之一；不可能同時已付又已退款
```

2) **讓架構「可被發現」**

一份短的系統地圖，勝過一堆散落 wiki。
如果最好的一版解釋只存在某個人腦袋裡，bus factor 其實已經是 1 了。

3) **把 incident review 當成「修心智模型」**

出事不只修 bug。
要修的是：為什麼大家會覺得那樣寫沒問題。

4) **把 LLM 當成 reader，不要只當 writer**

不要只用它來「生功能」。
也可以用來做：

- 「幫我解釋這個資料夾的 domain model」
- 「列出這個 service 看起來隱含了哪些假設」
- 「你會把哪些東西寫成 invariants / contracts？」

把模型當成工具，來幫你**稽核理解有沒有掉線**。

## 重點

技術債會讓你變慢。
認知債會讓你**迷路**。

如果你現在正在快速交付，很好。
但至少要確保你沒有默默刪掉「團隊能講清楚自己做了什麼」的能力。
