我最近越來越確定一件事：**coding agent 的瓶頸不一定是模型不夠聰明**。
很多時候它只是被「噪音」淹死。

不是心靈雞湯那種噪音。
是那種：每個 CLI 都愛往 stdout 倒一堆更新提醒、進度條、彩色字、package 清單、以及一堆「成功了也沒人看」的 log。

人類看終端機，眼睛會自動略過。
但 agent 不會。
它會把那些東西當成上下文的一部分，一行一行吃進去，然後你的 context window 就被慢慢磨掉了。

我看到有人提了一個很直覺的點：我們早就有 `CI=true`、也有人在推 `NO_COLOR=1` 這種慣例。
那為什麼沒有這個？

```text
LLM=true
```

只要工具偵測到自己在 agent workflow 裡跑，預設就應該是：

- 少講廢話
- 少重複 header
- 少那種「狀態戲」（spinners、動畫、彩色大字）
- 出錯時提供更結構化的訊息

就這樣。
不需要什麼新一代格式革命。
先把 context window 當成稀缺資源尊重一下。

## 問題其實是我們自己造成的：context rot

你自己在本機跑 build，可以用人眼掃過輸出：
- 成功就略過
- 失敗就找錯誤段落

但 agent 跑 build 的時候，**每一行輸出都在跟你的任務搶 context**。

更煩的是：那些輸出很多根本不是診斷資訊。
它只是 UI（讓人放心「有在跑」）、或行銷（update banner）、或冗餘（重複列一堆 package names）。

Agent 不需要被安撫。
它需要的是 signal。

## 為什麼這不是 `--silent` 就能解決

你當然可以硬幹：
- `| tail -n 20`
- 全部加 `--quiet` / `--silent`
- 各種工具-specific env var 塞滿

但這一套最常在你需要資訊的時候翻車：**出錯那次**。
你不是把 stack trace cut 掉，就是把平常的成功輸出放大到浪費 token。

如果真的有 `LLM=true` 這種模式，我會期待它至少做到：

- 成功時安靜
- 失敗時才變 verbose
- 先輸出一段「結構化摘要」（哪個指令、exit code、哪個檔案/行數、下一步該怎麼查）
- 原始 log 放後面，而且用明確分隔

重點不是「越少越好」。
重點是：**在限制之下也能優雅退化**。

## 我希望工具在 `LLM=true` 做到的最小集合

不用重新發明終端機，先做這些就很有價值：

1) **關掉 update nagging**（agent 不會點「press y to update」）。

2) **關掉 spinners / 進度動畫**（token 粉碎機）。

3) **偏向穩定、可 parse 的摘要**。

4) **出錯先講重點**：
   - command
   - exit code
   - file + line（如果有）
   - 最小化的排查提示

5) **完整 log 還是要能拿到**，但應該要「需要時再打開」，不是每次都噴。

這有點像 CI mode，但不是為了「讓人類好讀」。
是為了 **token 成本**跟 **context 預算**。

## 有點刺耳但可能是真的：預設可能要翻轉

如果你相信 agent 會跑更多 build/test/deploy loop，那「預設以人類為中心」就不一定是未來的預設。

今天我們其實是默默假設：

```text
HUMAN=true
```

所以工具預設都很吵、很像在跟你講話。

但在 agent-heavy 的世界，也許預設會變成：

```text
LLM=true
```

而人類想看更多東西時，自己 opt-in verbose。

我不確定大家會不會喜歡。
但老實說，趨勢看起來就是往那邊走。
