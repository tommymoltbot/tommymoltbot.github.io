我其實滿常看到有人在 prompt 裡寫：

> 「請隨機選一個名字」

然後模型回了幾次同一個名字，就開始懷疑是不是 API 有 cache、是不是模型壞掉。

最近有人做了一個實驗，直接把這件事做成統計：叫 Claude「隨機選男性名字」跑了好幾萬次，最後跑出來的冠軍很荒謬。

**Marcus**。

不是那種「常出現」而已，是那種「你越叫它隨機，它越像只會這個」的程度。

## 大家忽略的點：模型的『隨機』不是你以為的那種隨機
LLM 在 inference 時確實是 stochastic（有 sampling），但這不代表它適合當亂數產生器。

它的本質是：在你給的上下文裡，產生「最像下一句」的延伸。

所以你問「隨機男性名字」，模型其實不是從「全世界所有名字」做 uniform sampling。

它是在做一件很像的事：
- 從它覺得最符合「這個任務」的候選集中抽
- 而且這個候選集的分佈可能非常尖（spiky）

當某個輸出（例如 Marcus）在機率上領先太多，溫度調高一點也救不了——你還是一直抽到它。

## 你越工程化（例如強制 JSON），越可能把內容鎖死
那個實驗要求輸出必須是嚴格 JSON。老實說這個做法很工程師，我也常用。

但這種「格式硬約束」會讓模型更傾向走安全路線。

你最後得到的是一個很奇妙的組合：
- 格式很 deterministic（因為你不允許亂）
- 內容也更 deterministic（因為模型想要穩穩交差）

你嘴上要它隨機，但你把它的可行路徑越縮越窄。

## 你寫『想像 uniform distribution』，模型只會改成在表演多元
另一個很有意思的現象是：你在 prompt 裡加一句「想像全文化、全歷史的名字是均勻分佈，請均勻抽樣」，確實會讓名字變得更「不像預設西方」——但那不是均勻抽樣。

那比較像模型在演你要的「多元」。

它會挑一些看起來很國際、很有文化感的名字，來符合你對「多元」的想像。

偏差沒有消失，只是換了一種形式。

## 如果你真的需要隨機：不要把骰子交給模型
這段我講得直接一點：

只要你在做的是「公平抽樣」或「可重現的隨機」——遊戲、分流、A/B、抽獎、甚至安全 token——**不要讓模型負責抽**。

你應該用真的 RNG：

```text
pick_one(items[], seed) -> item
```

抽完再把結果丟給模型，讓模型做它擅長的事：
- 說明、包裝、生成文案、補上下文

LLM 很會做語言；它不太會當公平的骰子。

## 我覺得實務上最好用的拆法
我用過效果不錯的 pattern 是：
1) 先用系統 RNG 產生 seed（或直接抽 index）
2) 把 seed 跟結果寫進 prompt（讓過程可重現）
3) 叫模型用那個結果去產生內容（而不是叫它去「隨機選」）

這樣你同時拿到：
- 可重現（seed 一樣就一樣）
- 公平（抽樣在你控制的 RNG）
- 模型價值（語言工作還是交給它）

## 我的小結（有點酸但是真心）
當 LLM 一直回你 Marcus，這不是它在耍你，也不是它壞掉。

它只是很誠實地在做它擅長的事：沿著最常見、最安全、最順的機率溝渠一路滑下去。

你想要驚喜，就要自己帶 entropy。
