我最近越來越常看到「enshittification」這個詞。

有些人把它當成一種網路情緒：
「現在的網路就這樣啦，廣告一堆、訂閱一堆、功能越改越煩，你能怎樣。」

但我覺得它之所以刺耳，是因為它其實不是「品質下降」這麼簡單。

它比較像是一個很固定、很可預測的產品生命週期：

1. 先做出一個真的好用的東西。
2. 吸到大量使用者。
3. 開始從使用者身上榨價值（廣告、鎖定、誘導流程）。
4. 然後開始從依賴你平台的商家身上榨價值（抽成、規則改來改去、API 變貴）。
5. 最後產品變成「收費站 + UI」，功能只是為了把你留住或讓你多付錢。

挪威消費者委員會（Norwegian Consumer Council）剛出了一份報告：**《Breaking Free: Pathways to a fair technological future》**。
它我覺得厲害的地方是：沒有停在「大家都知道很爛」的抱怨，而是試著把這件事拆成「到底可以怎麼反制」。

我不是政策專家。我是那種會在週五晚上 debug 一個被「成長指標」搞爛的設定頁的人。
所以我讀它的角度很簡單：

> 如果你是做產品/做工程的，到底要怎麼理解「抵抗 enshittification」？

## 工程師該注意的一點：enshittification 通常是商業模式，不是意外

產品變爛的時候，我們很容易用「他們不會做產品」來解釋。

有時候確實是。

但更多時候，它不是做不好，而是「做得太對」——對 KPI 太對了。

- 讓你更難離開（switching cost）
- 讓每個使用者更容易被變現
- 讓分發更可控（逼你待在它的 app）
- 讓你更常回來（但不是因為你喜歡，而是因為你被設計得停不下來）

如果你有參加過那種會議：有人很自然地說「我們要不要讓取消訂閱更麻煩一點」，然後全場當作這句話很正常。
你大概就懂我在講什麼。

這份報告的論點是「系統性問題要用系統性方式修」。聽起來很大，但在工程世界其實超常見：
**爛的激勵機制，最後一定會變成技術限制與產品約束。**

你不可能一邊靠鎖定活著，一邊說「我們只想做個好產品」。

## 「不爽就換別的」在預設被設計過的世界裡，其實不太成立

報告（以及一起公開的公開信）有一段我覺得講得很直白：當市場上的選擇被這些東西扭曲時，「消費者選擇」就會變成笑話：

- 預設 app 是預載的
- 互通性被擋（interoperability）
- 匯出資料做得很敷衍
- 第三方 client 被 API 價格或政策打殘

從工程角度看，這些事情不是做不到。

你完全可以：
- 做好資料匯出
- 提供穩定 API
- 支援 federation 或至少支援第三方整合

但如果商業端覺得這些會降低鎖定，通常就兩種下場：
- 沒預算、不排進 roadmap
- 或是做了、上線了，然後「慢慢」被降級到形同虛設

## 最不舒服的鏡子：我們其實很擅長做出「對人不友善」的系統

我也不太想承認，但工程師對 enshittification 的機械結構真的越來越熟。

我們可以：
- 用 A/B test 去優化取消流程
- 用模型去預測誰要 churn
- 做出看起來像系統提示的彈窗
- 把真正重要的設定藏在三層頁面後面

技術上很強。
但價值觀上……有點怪。

公司內部最常見的說法是「別這麼戲劇化，這只是 optimization」。

但你把時間拉長看，結局其實很固定：
使用者越來越不信任產品，監管永遠慢半拍，最後網路變成一堆半殘服務，全部都想收你每月 9.99。

## 我覺得真的有用的幾個落點（就算你不是立法者）

這份報告主要是寫給政策制定者的，但我覺得有幾個點對「實作」層面很有意義。

### 1) 把 interoperability 當成可靠性（reliability）問題，而不是加分項

一個產品如果只能在你的 app 裡、用你的 UI、照你的規則才能完整運作，這不叫「整合得很緊」。
那叫「用設計包裝的脆弱」。

我自己會用一個很無聊但很有效的工程問題來測：

```text
如果我們公司明天消失，用戶能不能把資料拿走，而且拿走之後還是有用的？
```

如果答案是「我們有 PDF 匯出」——老實說那不是匯出，那比較像是告別信。

### 2) Switching cost 是產品債，而且會像利息一樣複利

鎖定一開始看起來像護城河，後來常常變成陷阱。

當你的留存靠的是摩擦而不是價值，你會同時怕競品、也怕自己的改動。
任何變更都可能觸發流失。

最後那種焦慮就會變成工程規範：
- 「設定頁不要動，會影響轉換」
- 「價格不要變簡單，怕大家發現其實不值」
- 「不要讓人太容易走」

你會越做越慢、越做越爛，但你會跟自己說這叫風險管理。

### 3) 只要你吃了生態系紅利，API 穩定性就是最低道德底線

這點我很有感。

如果一個平台靠第三方開發者把它做大，那把 API 當成抽租工具，本質上就是背刺，只是背刺寫在報表裡。

你不一定要開源、不一定要做公益。
但你必須選清楚你的平台到底是：

- 共享基礎建設
- 還是 bait-and-switch

## 最後一個很現實的問題：這趨勢真的能反轉嗎？

我不覺得我們會回到「早期很可愛的 web」。

但我也不相信 enshittification 是宿命。

更像是：**只要我們持續獎勵「收費站化」的公司，我們就會持續得到收費站化的產品。**

所以政策當然重要。

但還有一個更小、更立即的東西也很重要：工程師對自己正在做的東西誠實。

不用上綱。
只要在心裡、在團隊裡把它叫對名字。

你一旦開始把 dark pattern 叫做 dark pattern，就比較難再躲在「engagement」這種字眼後面。

很多改變其實就是這樣開始的：不是靠宣言，而是某個團隊默默決定「這次不要再做一個更惡意的流程」。
