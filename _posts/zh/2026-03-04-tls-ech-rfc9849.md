ECH（Encrypted Client Hello）拿到 RFC 編號（RFC 9849）這件事，表面上很無聊，但它其實是在慢慢改變「網路隱私的基準線」。

以前我們跟非網路圈的人解釋 HTTPS 時，常常要補一句：

> 「內容有加密，但你連到哪個網站，很多時候是看得到的。」

那個最直觀的破口，基本上就是 TLS 的 `ClientHello`。ECH 就是要把這個洞補起來。

但我也不想把它吹成魔法。ECH 不是讓你在網路上隱形，它比較像是：*把一個很吵、很好收集、很好變現的 metadata，硬生生降低可觀測性*。

## ECH 到底在藏什麼？為什麼重要？

TLS 連線一開始，client 會送 `ClientHello`。傳統上裡面會包含 SNI（Server Name Indication）——也就是你要連的網域。

所以就算你用 HTTPS，被動旁觀者還是常常能看到類似這種東西：

```text
ClientHello:
  server_name = "example.com"
  alpn = ["h2", "http/1.1"]
  ...
```

ECH 的核心設計是：

- 你還是會送一個明文的 **ClientHelloOuter**（因為封包得先「路由到正確的前門」）
- 同時再送一個加密的 **ClientHelloInner**（真正的 SNI 跟一堆敏感 extension 都在這裡）
- server 如果解得開就用 inner；解不開就 fallback

這個 outer/inner 的拆法，才讓 ECH 有機會在 CDN、多租戶這種現實世界裡部署。

## 五個「真的會不一樣」的地方

### 1) 它不再是你可以永遠拖延的 draft

IETF draft 可以拖很久（真的很久）。RFC 編號不代表明天大家都開 ECH，但通常意味著：

- 廠商比較願意把它排進 roadmap
- 企業比較不會把它當成「實驗性怪東西」
- 周邊工具跟文件會開始長出來（debug、觀測、合規用詞）

ECH 需要 client + resolver + server/CDN 多方協同，標準化能降低「會不會又改？」的焦慮。

### 2) 隱私收益是真的，但範圍很明確

ECH 主要針對的是那個最容易被 log、最容易被拿去賣的訊號：

-「這條 TLS 連線是要去 *哪個 hostname*？」

但它不會自動幫你藏掉：

- server 的 IP（除非你在共享前門後面）
- 流量形狀（大小/時序仍然可以被做流量分析）
- DNS 查詢（如果你 DNS 走明文，照樣會漏）

所以如果你的威脅模型是「咖啡廳 Wi‑Fi 想看你逛哪個站」，ECH 是大加分。

如果你的威脅模型是「能跨網路做流量關聯的對手」，ECH 有幫助，但不是句點。

### 3) 逼你承認：TLS metadata 不是免費的

很多 production 系統其實默默依賴握手階段的資料：

- 用 SNI 做 routing / tenant selection
- 用 metadata 做流量分類
- 某些安全設備想在加密前就「先知道你是什麼」

ECH 不會把需求消滅，但它會把事情往上推：

- 你是不是該把 routing 放到 CDN edge，而不是企業內網的中間盒？
- 你是不是要把策略放到 endpoint（client agent）而不是旁路監聽？
- 你要不要接受「被動觀測 hostname-level logging」會越來越難？

這跟 TLS 1.3 的方向一致：旁觀者少拿到一些東西，endpoint 要多負責一些。

### 4) 操作層面：ECH 是「DNS + TLS」，痛點也在這裡

ECH 要靠 DNS 發現 ECH config（通常走 HTTPS/SVCB record），而且理想上你希望 DNS 本身也是走加密的傳輸（不然你把 SNI 藏起來，DNS 又漏掉了）。

真的會炸的通常是很無聊的事情：

- DNS cache 太久、資料過期
- split-horizon DNS（內外回不同答案）
- 中間盒對 TLS 握手動手動腳

ECH 的 outer/inner 設計有比較「可恢復」的失敗模式，但可恢復仍然代表：

- 有些情況會 fallback
- 有些情況會 retry
- 觀測指標會變得更難解釋（尤其是 latency 和 retry rate）

如果你有線上服務，我會建議你至少盯：

- 握手成功率（區分有/沒有 ECH）
- retry 比例 / latency spike
- 地區性問題（常常是 DNS 基礎設施差異）

### 5) 工程師需要更新「HTTPS 到底藏了什麼」的心智模型

我還是常看到安全文件寫：

- 「TLS 會把所有東西都加密」（錯）
- 「SNI 是加密的」 （過去大多是錯；現在變成「看情況」）

ECH 讓正確的描述變得很煩，但更接近現實：

- *如果 client 支援 ECH、DNS 路徑能提供 ECH config、server/CDN 也支援 ECH，那 hostname 才有機會對被動旁觀者隱藏。*

對，這句很長，但這就是現實世界。

## 如果我手上有 production service，我會怎麼做

- 你在大 CDN 上：先確認 CDN 的 ECH 支援狀況與 rollout 計畫。
- 你自己終止 TLS：老實評估你能不能在不破壞 routing / 證書選擇的前提下支援 ECH。
- 更新 threat model 與文件：不要在說不清前置條件的情況下承諾「訪問網站不可見」。

另外，如果你正在做任何「靠網路識別碼當作假名」的產品假設，我會先把這個假設打個問號。

協定層的隱私改進通常只會往同一個方向推進：讓被動收集變得更困難。
