![Power user 正在慢慢死掉](/img/posts/2026-02-25-slow-death-of-the-power-user-01.webp)

我今天看到一篇文章在講一件很直白、甚至有點刺耳的事：**power user 正在消失，而產業還在替這件事鼓掌。**

我一開始也很想把它當成「又來了，下一代不懂電腦」的老梗。但讀到後面，我覺得重點其實不是年齡。

重點是：這 10–15 年「主流計算」的形狀變了，而且它會主動獎勵你當個消費者、懲罰你當個懂工具的人。

更麻煩的是：你就算不是 power user，你最後也會付這張帳單。

## 先講清楚：power user 不是熱愛終端機的人

很多人以為 power user 就是：鍵盤快捷鍵背很熟、會寫 shell script、看不起 GUI。

不是。

我對 power user 比較務實的定義是：
- 看到錯誤訊息會讀，會想弄懂它在講什麼
- 用一個陌生系統，願意花 20 分鐘建立心智模型
- 東西壞了會 debug，不是第一時間怪「這產品爛」
- 能把幾個工具串起來做小自動化，不用每次都等平台賞 API

這種人最後會變成什麼？

他們會變成「大家都在用但沒人付錢」的基礎建設：寫教學、提 issue、給 reproducible bug、在論壇回答問題、做那些把產品從 60 分推到 80 分的雜事。

你把 power user 砍掉，本質上是在砍斷一條知識供應鏈。

## 這不是意外：籠子本來就設計得很舒服

我不覺得這是單純的「世界變簡單了」。

主流平台這些年一直在往「家電化」的方向走：
- 安全（部分是真的安全，部分是行銷用語）
- 方便
- UX 可控
- 商業變現

這些 incentive 合在一起，會把使用者推向一個狀態：
> 你可以做平台預期你做的事，但你很難做平台沒預期的事。

而且更微妙的是，平台會逐步把「學會系統怎麼運作」的那幾個入口藏起來：
- 檔案系統變得不可見、不可預期
- 安裝非官方軟體會被不停警告
- app 之間很難互通
- 背景程序被各種限制

你如果從小就在這樣的環境長大，你當然不會學到「可組合性（composability）」。

而 debug 很大一部分其實就是：**沿著系統的接縫把問題追下去。**

現在的產品就是在拼命把接縫抹平。

## 更諷刺的點：AI 助手可能讓這件事更快

我不是反 AI。只是我覺得這裡有個坑。

如果主流工作流變成：
> 「講意圖 → 得到輸出 → 上線 → 不追問中間發生什麼」

那我們會養出一種新型態的「power user」：很會 prompt，但不會 debug。

這種能力其實很脆弱。

因為當 AI 失手（一定會）時，你需要的不是更漂亮的 prompt，你需要的是：
- 看 log
- 懂網路邊界
- 知道效能 tradeoff
- 知道資料到底去哪了

這些不是老技能，這些是讓系統「可讀」的技能。

## 我自己的結論：把「可讀性」當成一種工程資產

這不是道德勸世文。這是 operational 的問題。

你如果是工程團隊、或你在做產品，你最後都會遇到很無聊但很真實的問題：
- 半夜出事，誰真的能把問題找出來？
- vendor 改 policy，你是恐慌還是能繞過去？
- 你只有一條「官方 happy path」嗎？還是你有第二條路？

很多公司其實是在賭：平台會處理。

可以啊。直到平台不處理。

## 不英雄也能做的幾件小事

1) **保留至少一條「真電腦」工作流**
   - 看得到檔案、跑得動工具、知道東西放哪裡。

2) **把 debug 當成一級技能在教**
   - 不只教框架怎麼用，也教怎麼觀測系統在幹嘛。

3) **獎勵好奇心，不只獎勵產出**
   - 能出貨很重要，但如果沒人能解釋系統，你是在幫未來的 outage 埋雷。

4) **優先選「保留逃生門」的工具**
   - 好產品不是永遠把複雜度藏起來，而是讓你可以逐步拿回控制權。

我不浪漫化以前。以前的 power user 文化確實常常很毒、很愛 gatekeeping。

但「使用者可以擁有工具、可以理解工具」這件事本身一點也不尷尬。

我寧願保留這條路，也不要走到最後只剩下：
> 「我知道哪個訂閱方案可以解鎖匯出按鈕。」
