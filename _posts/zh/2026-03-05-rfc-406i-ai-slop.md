我今天點到一個連結，直接被一個「像 HTTP error」的頁面嗆爆：**AI slop detected，拒收。**

那個網站把它叫 *RFC 406i*，說是「一套標準協定」用來處理、丟掉那些低品質的 AI 生成 PR / issue / security report。

很明顯是惡搞。

但老實說，它也是我最近看過最誠實、最貼近 maintainer 心情的文字。

## 問題不是 AI，本質是「努力不對稱」

他們真正討厭的不是「你用了 LLM」。

他們討厭的是這個交易結構：

- 你花 **30 秒**生出一段看起來很專業的 PR 描述
- 我花 **3 小時** 才能確認你講的到底有沒有一句是真的

這個落差就是現在的災難。

你只要維護過任何 repo（開源或公司內部），就懂那種感覺：

PR 文案寫得很漂亮、邏輯很順、最後還來一句「robust and scalable solution」。

然後你拉下來跑，發現作者根本沒跑過一次。

AI 沒有發明這件事。

AI 只是把「把別人時間當垃圾桶」這件事變得更便宜、更大量。

## 「能 compile」根本不是門檻

406.fail 那篇寫得很毒，但有句話我很認同：語法、排版、甚至 CI 綠燈，都只是地板，不是天花板。

因為現在你很難保證：

- PR 的 CI 綠燈，是不是因為它順便把測試改成只驗證 `True == True`

更深層的問題是 **責任歸屬**。

一個人丟進來的變更，應該要能回答：

- 原本到底壞在哪？
- 你改了什麼？
- 為什麼這樣就修好了？
- 你犧牲了什麼 tradeoff？
- 下週會不會又壞回去？

LLM 可以生成「看起來像答案」的文字。

但「像」不是「是」。

## 真正能用的 anti-slop protocol 長怎樣

如果把 406.fail 的情緒抽掉，我覺得 maintainer 真正在要的是一套很無聊、但會逼你把努力對齊的規則。

### 1) 逼你交可重現的問題
Bug report 至少要有：

- 明確版本 / commit
- 最小可重現範例（minimal reproduction）
- expected vs actual
- log / stack trace

不能重現，就不用談修。

### 2) 逼你把 diff 縮到最小、最能證明點的程度
LLM 生成的 PR 很容易 scope creep。

成熟的 review 流程反而要：

- 變更範圍很緊
- 先放 failing test / 可重現案例
- 讓修正「一眼看得懂」

### 3) 逼你把 end-to-end 的故事講清楚，而且你要能負責
你開一個 PR，其實是在說：

> 「我有跑過，我理解這個改動。你追問我我答得出來。」

如果你答不出來，只能說「AI 想的」，那不是 contribution，那是把工單丟到 repo。

### 4) 明講什麼東西會被丟掉
我其實很支持 repo 放一段簡短的「拒收模板」（不用像 406.fail 那麼毒），大概像這樣：

```text
感謝提交。

我們無法 review 缺乏驗證的 AI 生成變更。請至少補上：
- minimal reproduction 或 failing test
- root cause 的清楚說明
- 你自己跑過並驗證修正的證據

補齊後再送一次。
```

重點不是羞辱，是避免 maintainer 變成「LLM output 的免費驗證服務」。

## 尷尬的是：這件事也會先炸在公司內部
大家都在講 open source，但我覺得公司內部更快遇到：

- 「AI 寫的」變成沒人能 debug production 的原因
- 「AI 寫的」變成不寫 runbook、不寫設計決策的藉口
- PR 文字看起來很完整，但你問一個 follow-up，沒人回答得出來

等到服務真的有 user，你最後一定會遇到：

- pager
- 亂七八糟的 incident timeline
- 還有一句很致命的問題：「這誰 approve 的？」

## 我的結論

我不反 LLM，我自己也用。

但我很反對「把一堆沒人願意負責的文字丟進系統裡」。

如果工具讓你寫字變快 10 倍，那 verification 的門檻就應該變高，而不是變低。

所以你看，一個惡搞到不行的 meme-RFC，反而把這件事講得最準。
