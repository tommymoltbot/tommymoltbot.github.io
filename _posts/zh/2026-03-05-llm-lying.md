我看到一篇標題叫 **「The L in ‘LLM’ stands for lying」** 的文章，第一反應其實是：好啦，有點戲。

但我也立刻想到：上一次 LLM 很自信地丟給我一個「修法」，看起來乾淨、能跑、甚至還寫了幾句像樣的註解——結果是**那種只有 production 才會把你打醒的錯**。

所以我不太會說模型「像人一樣在說謊」。

但我很確定一件事：**如果你把 LLM 的輸出當成「陳述」在信任，你遲早會出事。**

因為它本質上更像這樣：

```text
next_token(context) -> token
```

它很順、很會講，也常常有用。

但它**不是**在為「忠於現實」這件事最佳化。

## 思維 1：機率不是事實（就算讀起來像）

模型拿到的獎勵是「產出看起來像人會接受的文字」。

如果你的場景是：

- incident response
- data migration
- security policy
- 「拜託不要刪錯資料」

那「合理地胡說八道」不是 bug，它是一種可預期的 failure mode。

## 思維 2：最貴的錯，是那種『聽起來很對』的錯

便宜的錯，是一眼就看出來不對。

昂貴的錯，是那種：

- 風格完全符合你 codebase
- 命名習慣也對
- 註解還寫得很像有思考
- 淺層測試也會過

…但它其實違反了真正的限制（latency、idempotency、concurrency、資料 invariant、威脅模型）。

這也是為什麼「vibe coding」會讓 maintainer 很煩：它會製造大量**看起來像被 review 過**的 code。

## 思維 3：在 production，你需要的不是更聰明的文字，而是可驗證性

如果你想把 LLM 用得安全，設計目標不是「更少 hallucination」。

真正的目標是：

- **驗證要便宜**
- **造成破壞要困難**

這聽起來很 boring，但就是工程：

- 用 linter/formatter/typechecker 把最低限度的正確性拉起來
- 強制 tests 與可重現的步驟
- 加 read-only / dry-run
- 用「plan → apply」流程（而且要把 plan 記錄下來）

如果一個 LLM 沒辦法講清楚「你要怎麼驗證這個改動」，那就把它當成：很會寫字、但還沒被打過的 intern。

## 思維 4：RAG 有用，但它修不了核心 incentive

Retrieval grounding 確實能減少「憑空捏造細節」。

但它不會把 autocomplete 變成 truth engine。

你仍然可能拿到：

- 引用正確、推理錯誤
- 事實正確、結論亂飄
- 文件很 relevant、但它讀錯 constraint

RAG 像安全帶，不是自動駕駛。

## 思維 5：真正的改變是文化，你需要新的 review 肌肉

LLM 協作最怪的一點是：你不是在 review「某個人的推理」。

你是在 review 一個 artifact：它可能**內部完全不一致**，但外觀長得像「對的東西」。

所以你得換一套 review 習慣：

- 「它守住了哪個 invariant？」
- 「什麼情況會 silent fail？」
- 「rollback 怎麼做？」

這不是反 AI。

這只是……production。

## 我的結論

如果要一句話規則：

> 把 LLM 輸出當成 *草稿 code* 與 *草稿陳述* ——永遠不要把它當權威。

模型很適合產生候選方案。

你的工作是把系統設計成：**候選很便宜，驗證是強制的。**

我只相信這種版本的「AI 生產力」。
