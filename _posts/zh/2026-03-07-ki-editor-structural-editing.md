有一種「編輯器點子」很常見：每隔幾年就會被拿出來講一次，會有一小群人瘋狂愛上，然後就……沒有然後了。

**結構化編輯（structural editing）** 就是這種。

核心想法很直白：程式碼本質上是 AST（一棵語法樹），那你為什麼要一直假裝它只是一段字串？

我真正想要的是：移動、選取、變形時，操作的是 **語法節點**，而不是在那邊拉選取範圍、數逗號、祈禱括號不要爆。

Ki Editor 是我最近看到，少數把這件事當真做的專案，而且它多加了一個很有攻擊性的組合：**多游標 + 結構操作**。

## 為什麼結構化編輯一直沒「贏」
如果你用過 Lisp 的 paredit，或玩過 tree-sitter aware 的 motion，你一定知道那種爽感：

- 刪參數不會把括號弄壞
- 選取「下一個語法單位」不用算逗號
- refactor 像在搬動「意義」，不是搬動字元

但現實是，大多數工程師最後還是回到一般文字編輯。

我自己的猜測（偏第一線工程師視角）：結構化編輯常常輸在它要求你為每個小動作都換一套腦內模型。

「更正確」還不夠。
它必須在你原本的肌肉記憶之上，真的 **更快、更順**，不然你會直接跳船。

## 真正的關鍵：選取（selection）其實是 API
Ki Editor 在首頁那幾句話，其實很誠實地透露了它在乎什麼：*multi-cursor combinatoric modal editor*。

聽起來很像行銷，但我覺得背後是個正確方向：

- modal editing 不只是 normal/insert 這種二元切換
- 它應該是一套 **一致的選取系統**

如果「選取」在 word / line / syntax node 上都能用同一套邏輯，很多操作會突然變得不需要特別記。

你不用學 40 個「特化」的 refactor 指令；你學的是少數幾個操作，然後靠組合去完成更多事。

結構化編輯要落地，我覺得就得用這種語言。

## 多游標 + AST 操作：危險，但很香
多游標本來就很強，但也很容易濫用。

當多游標只在字元層級運作，你很容易做出「同步且完美的垃圾修改」。

但如果多游標可以在 **語法節點** 層級運作，事情會變得很有意思：

1) 你可以在很多地方做同樣的變形，同時保證結構不被破壞。

2) 你可以把 refactor 變成「語意上的批次操作」。

差別大概像這樣：

- 「希望我沒有漏掉某個逗號」
- vs 「我把所有 call site 的第二個參數都換成另一個 expression」

如果 Ki Editor 真能把這件事做得可靠，那它不是小眾玩具而已。

## 我最在意的質疑：逃生口在哪？
任何「嚴謹」的系統，都需要一個務實的出口。

有時候你就是想直接改 raw text。
有時候你在寫 parser 不懂的檔案。
更多時候是：你正在打草稿，你的 code 根本還沒寫完，AST 一定是 invalid —— 因為你還在想。

會贏的 editor 不是那種說「你永遠不要碰文字」的。
而是能做到：

- **結構有效時給你力量**
- **結構無效時不要懲罰你**

「不要懲罰使用者」這點，我覺得是很多野心型編輯器的死亡線。

## 為什麼我現在在意（對，因為 AI）
LLM 正在把我們推向更高階的程式碼操作：

- 生成 code
- 大段 refactor
- 在模組之間搬邏輯

在這種世界裡，一個只懂字元的文字編輯器，會越來越像底層工具——不是不能用，而是你會開始覺得「這也太 low-level 了吧」。

結構化編輯不是為了讓人打字更快。
它更像是為了讓人 **更快地 review / reshape** 程式碼。

而如果你真的在 production 做過事，你大概也同意：最後卡住你的，多半不是「打字」，是「驗證與審查」。
