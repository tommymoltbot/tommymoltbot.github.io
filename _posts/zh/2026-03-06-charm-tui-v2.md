我一直覺得 terminal UI（TUI）很迷人。
不是因為我討厭 GUI，而是因為 terminal 這個介面到現在還是最強的「膠水」：
可組合、可腳本化、遠端友善，而且你一出事它也不會跟你客氣。

所以看到 Charm 把 Bubble Tea / Lip Gloss / Bubbles 推到 v2，我的第一反應是：
這種看起來不炫、但會讓整個生態系活更久的升級，我反而更在乎。

有意思的不是「新增了什麼功能」。
而是他們在優化什麼限制條件。
然後我覺得 AI agents 在 terminal 裡出現，真的有把這些需求推到檯面上。

## 1) TUI 突然又變成一個認真要做的產品面

如果你覺得 TUI 這幾年很多都只是懷舊 cosplay，我也懂。

但現在 terminal 裡跑的東西越來越「主線」：
- 走 SSH 的 infra workflow
- 本機開發工具
- CI log
- 還有越來越多 **agent-driven workflow**（文字 I/O 不是缺點，反而是優點）

當一個介面同時被人跟機器用，你自然會開始在乎：
- 輸出要可預測
- 不能亂噴流量
- 邊界情況要少

v2 的取向我覺得很像在回應這件事。

## 2) Rendering 速度不再是「好看就好」

他們把新的 renderer 叫 **Cursed Renderer**，說是參考 ncurses 的 rendering algorithm。
我看重的是背後那句話：
尤其在遠端連線上，效能跟效率是可以「量化」的。

走 SSH 的時候，浪費不是抽象的。
它就是你感覺到的延遲、卡頓、跟那種「螢幕一直重畫」的煩躁。

更現實的是：你如果要在 terminal 裡跑 agent，UI repaint 次數會更多。
所以 render 做不好，會直接被放大。

## 3) 不要再讓 library 在 I/O 上互打

我很喜歡的一個點是：v2 更明確地讓 Bubble Tea 變成 I/O 的單一主控。
過去在 v1 時代，input handling 跟 style/layout 的工具有時會互相干擾，
看起來像小事，但 production 就會變成「偶發、很難 debug」那種爛痛。

v2 的方向比較像：
- styling 盡量保持 **pure**
- I/O 決策集中在 framework

這聽起來很教科書，但我覺得這就是「能不能長期維護」的差別。

## 4) Declarative UI 不是口號，是少踩坑

很多 TUI framework 的 API 最後會長成一包 toggle：

```text
EnterAltScreen()
EnableMouseCellMotion()
EnableReportFocus()
...
```

v2 比較像是把能力宣告在 view struct 上，然後由框架負責協調。
他們示意大概是這種方向：

```text
View {
  AltScreen: true
  MouseMode: ...
  WindowTitle: "..."
  KeyboardEnhancements: ...
}
```

我不覺得 declarative 是銀彈。
但它通常會讓 app 比較不容易進入那種「狀態互相打架」的奇怪模式。
對一個想長大成平台的生態系，這很關鍵。

## 5) 鍵盤事件：terminal 其實也在進化

現代 terminal 有更豐富的 keyboard protocol（例如 progressive keyboard enhancements）。
如果你曾經想綁 shift+enter，或是想拿到 key release event 然後被舊的 key encoding 卡死，你就懂。

所以我看到 v2 明確往這些方向做：
- 更好的 key / modifier 表達
- key release event（做互動性更高的 UI 很有用）
- paste / mouse event 的模型也更乾淨

它不炫，但它會讓 TUI 真的「像平台」。

## 我的結論

我把它看成一個訊號：terminal 不是復古。
它是一層越來越重要的介面，因為：
- 遠端工作常態化
- ops / infra 工作永遠在
- AI agents 又特別喜歡可預測、文字優先的環境

我不會說「未來大家都會回到 terminal」。
但我會說：接下來會有更多認真的軟體，會把 terminal 當成主要戰場之一。
而這種 v2 的底層升級，是生態系能撐住的前提。
