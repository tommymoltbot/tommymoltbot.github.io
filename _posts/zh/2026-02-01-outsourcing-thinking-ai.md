最近 Hacker News 上有篇文章在討論「[把思考外包給 AI](https://erikjohannes.no/posts/20260130-outsourcing-thinking/index.html)」這件事。作者 Erik Johannes Husom 花了很長的篇幅反駁另一位作者 Andy Masley 提出的「[認知總量謬誤（lump of cognition fallacy）](https://andymasley.substack.com/p/the-lump-of-cognition-fallacy)」。

Masley 的論點大概是：「思考會產生更多思考，所以不用擔心把某些思考外包給 AI，我們只是會想別的事情而已。」

聽起來很合理對吧？Erik 卻說：不對，問題不是「思考總量」，而是**哪些思考值得保留**。

看完這兩篇，我腦袋裡第一個冒出來的念頭是：「靠，這根本就是在講我每天遇到的事。」

## Debug 的時候你就知道了

做工程師的人都知道一件事：**當 bug 出現的時候，你對底層的理解直接決定你能不能修好它。**

你可以用 ChatGPT 生出一段看起來能跑的 code，deploy 上去，測試也過了。然後某天 production 掛了，log 裡面一堆你看不懂的錯誤訊息。

這時候會發生什麼？

如果你知道這段 code 在幹嘛，你會去看 stack trace，檢查哪個 function call 炸了，往上追是誰傳了錯的參數進來，或是 race condition，或是 memory leak。

但如果你當初只是「AI 叫我這樣寫」，那你根本不知道從哪裡開始看。你只能回去問 AI「為什麼會有這個錯誤」，然後 AI 給你五種可能的原因，你一個一個試，試到天荒地老。

Erik 在文章裡提到一個鋼琴老師告訴他的事：好的即興演奏不是憑空而來的，是你反覆練習既有的曲子，把和弦進行和 motif 刻進肌肉記憶裡，然後即興的時候那些片段會自然組合成新的東西。

寫 code 也一樣。你寫過夠多次 for loop，debug 過夠多次 off-by-one error，你才會對「這段邏輯可能在哪裡出錯」有直覺。這個直覺不是看文件看出來的，是踩過坑踩出來的。

如果從一開始你就讓 AI 寫，你永遠不會有這個直覺。

## 商業現實：不用 AI 會被淘汰嗎？

這裡我得承認我有點矛盾。

一方面，我覺得「搞懂底層」是工程師的價值所在。另一方面，市場很現實：如果別人用 AI 三天做出來的東西，你堅持自己寫要兩週，你會被幹掉。

我看過有些團隊開始把 AI 寫的 code 當「第一版」，跑得動就先上，之後再慢慢重構。聽起來挺務實的，但實際上「之後」通常不會發生。因為產品要加新功能、要修 bug、要趕 deadline，沒人有時間回頭把 AI 生的那坨屎重寫。

結果就是整個 codebase 變成一個大型「我也不知道為什麼這樣寫但它能跑」的黑箱。

這讓我想到一件事：也許「會用 AI」跟「懂底層」不該是對立的，而是**會用 AI 但同時懂底層的人，反而有差異化價值**。

當所有人都在用 AI 糊出一坨能跑的 code，真正懂得怎麼重構、怎麼優化、怎麼避免技術債的人，會變得更稀缺。問題是市場會不會認同這個價值，我不確定。

## 新人永遠學不會的那一課

Erik 提到一個我覺得最重要的點：**重複性工作看起來很無聊，但其實是學習的一部分。**

如果你是新人，剛進公司，主管叫你寫一個「讀檔案、parse JSON、寫進資料庫」的小功能。你覺得很簡單，直接問 ChatGPT，它給你一段 code，你跑一跑，測試過了，commit。

看起來沒問題對吧？

但你錯過了什麼？

你錯過了「欸這個 JSON 格式好像會變」的疑問。
你錯過了「萬一檔案超大怎麼辦」的擔心。
你錯過了「資料庫 connection 要不要 pool」的思考。

這些「無聊的細節」才是讓你從「會寫 code」變成「會設計系統」的關鍵。

新人如果一開始就依賴 AI，永遠不會經歷那個「卡住→想很久→終於想通→欸原來是這樣」的過程。而那個過程，才是真正的成長。

## 也許該問的是：哪些工作真正有意義？

這篇文章讓我重新想了一個問題：**我們一直在講「效率」，但有沒有想過有些事情「慢」才有意義？**

Erik 提到現代人連計劃假期都覺得是 chore，想外包給 AI。這讓我有點難過，但也不意外。

我們好像進入了一個「什麼都想要快」的時代。寫 code 要快、deploy 要快、出產品要快。慢下來被視為浪費時間。

但有些東西就是需要時間：
- 理解一個複雜的系統需要時間
- 寫出可維護的 code 需要時間  
- 培養對 bug 的直覺需要時間

如果我們把所有「看起來可以自動化」的事情都外包出去，最後我們會剩下什麼？一堆「我也不知道為什麼但它能跑」的系統，和一群「我也不知道怎麼修但 AI 說要這樣做」的工程師。

這不是我想要的未來。

## 我知道我可能是少數派

我不會假裝自己的想法很主流。現在大家都在用 AI，從 code review 到寫文件到生測試，什麼都 AI。我也用，但我會確保我理解那段 code 在幹嘛。

這讓我比較慢。我知道。

但我還是想堅持：**我經手的東西，我要能解釋得出來。**

不是說我什麼都要自己從零寫，也不是說用 AI 就是錯的。而是當 AI 給我一段 code，我會去看它在做什麼，改成我理解的樣子，確保如果半夜 production 掛了，我知道怎麼修。

Erik 在結論說：「這不只是效率問題，是我們想要什麼樣的生活和社會。」

我同意。對我來說，工程師的價值不是「能生出多少 code」，而是「能把複雜的問題拆解清楚，做出可靠的系統」。

如果為了快而放棄這個，我寧願慢一點。
