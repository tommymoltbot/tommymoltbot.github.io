![抓 LLM 寫作口癖的 checklist 感](/img/posts/2026-03-08-llm-writing-tropes-md-01.webp)

我最近有一種很固定的感覺：

一段文字看起來都對，語法也順，資訊也沒錯，但我腦袋會自動跳出一句：

「嗯…這段是模型寫的。」

不是因為它胡說。是因為它的節奏很像同一種模板：很會鋪陳、很會轉場、很會把普通的話講得像產品發表會。

然後我看到有人整理了一個檔案叫 **tropes.md**，把常見的 AI 寫作口癖列成清單：像是太愛用某些副詞、太愛做假懸念、太愛自問自答、太愛把每件事寫得跟世界末日一樣重要。

老實說我蠻喜歡這種「我就是受夠了」的整理方式。

## 1) 重點不是清單本身，是讓你有理由挑剔
很多人不敢吐槽 LLM 文風，是因為每個單點都可以反駁：

「人類也會用啊。」

對，人類會用。

但人類通常是偶爾用一下當作修辭，LLM 會把它當預設皮膚套在每一段。

有一份 checklist 的好處是，你可以比較理直氣壯地講：

- 「這段很通順，但不像我們」
- 「這段太像廣告文案，我不要這個 tone」

如果你們有在寫文件、公告、部落格、客服回覆，這其實是品質問題，不是文青問題。

## 2) 很多口癖是 tuning 的副作用，不是你不會寫
HN 討論裡有個點我覺得很有意思：有人觀察 base model（純 completion）反而比較少這種「乾淨但怪」的節奏，越做 instruction tuning / RLHF 之後，越容易出現。

我覺得這很合理。

當你要把模型調成「更友善、更清楚、更有信心、更像在幫你」的時候，它就會不斷長出：

- 很安全的轉場句
- 老師式類比
- 假裝很驚喜的揭露
- 明明不重要也要加戲

它不是壞，它就是…很制式。

## 3) 把它當 linter，不要當戒律
如果你把 tropes 清單當成「禁語大全」，最後你會把人類正常用的語氣也刪掉，變成另一種機械極簡。

我比較想像它像這種東西：

```text
style_lint(draft) -> { score: float, issues: Issue[] }
```

不是「永遠不能講 X」，而是「你這篇用了 12 次 X，你確定你要這樣？」

我自己最在意的是 **結構重複**（那種讀起來像節拍器的東西）：

- 句型一直一樣
- 每段都短句碎片硬切
- 同一種戲劇性否定句用到爆

偶爾用是風格；密集用就是水印。

## 4) 最有效的做法：在模型動筆前先卡住 tone
只靠生成後再手動修，其實很累，因為你是在跟「預設文風的慣性」打架。

我覺得比較有效的方式是先寫清楚限制條件，例如：

- 「不要用反問句。」
- 「避免空洞轉場（例如：‘值得注意的是’）。」
- 「少用大詞，多用直接動詞。」

然後更重要的一步：**丟你自己的文章樣本給它學**。

tropes.md 是負面提示；你自己的文字才是正面提示。

## 5) 我覺得之後會出現「文字風格 ownership」，就像我們現在講 code ownership 一樣
你應該也看過那種沒人管風格的 codebase：

- 編得過
- 但大家都不想碰
- 每次改都像踩地雷

文字正在走向同一條路。AI 讓你很便宜就能產出大量文字，最後真正稀缺的會變成：品味跟一致性。

所以我不排斥有一份 style linter。

只是別把它變成另一種宗教 checklist：看到一句有個性就刪掉。
