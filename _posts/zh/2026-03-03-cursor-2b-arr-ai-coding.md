TechCrunch 報導：Cursor 傳出年化營收（ARR）已超過 **20 億美元**。

我第一個反應不是「AI 要取代工程師了」。我腦中比較像是：*好，這已經不是工具愛好者的選擇題了，這是採購題。*

一旦某個類別長到這種規模，重心會從「個人開發者覺得爽不爽」變成「企業預算、合規、導入成本、可量化 ROI」。而這會把整個市場的玩法改掉。

## 1) 20 億 ARR 的意思：這不是玩具市場

ARR 這種算法當然粗，但它有一個很實用的功能：讓你判斷「這到底是不是小圈圈自嗨」。

如果這個數字大方向沒錯，代表背後要嘛是：

- 超多 seat（席位）
- 超多企業方案
- 或兩個都有

到了這階段，最大的對手不是「另一個更酷的編輯器外掛」。最大的對手是：**CFO 的眉頭**。

因為一旦變成固定支出，問題就會變成：*你到底買到了什麼？*

## 2) 個人開發者會跳槽是真的，但對企業來說不一定重要

文章也提到，最近有人在 X 上炒「Cursor 動能是不是在掉」，原因是一些個人開發者轉去 **Claude Code**。

我完全相信。開發者的忠誠度本來就很低（這句我當稱讚）。我們會因為一個 feature 換工具，然後再用兩個週末把設定檔搬家，還覺得自己很浪漫。

但企業不是這樣。

企業更在乎的是：

- 供應商風險
- 資安與資料處理方式
- 稽核與可追溯性
- 集中帳單與管理
- 支援 SLA

所以就算某個工具對一部分 power user 沒那麼爽，只要它更容易被 IT/資安/採購接受，照樣可以在公司內贏。

老實說這點我有點不舒服：最後贏的產品，可能是「讓 IT 好管理」的那個，而不是「讓工程師真正好用」的那個。

## 3) 真正的護城河：不是模型比較強，是你把工作流吃進去了

模型品質當然重要。但當大家都能接到強的 foundation model 之後，長期護城河會長得更像：

- 你跟工作流黏多深
- 你看得到多少 context（repo、ticket、log、runbook）
- 你累積了多少「公司專屬」的知識

這就不是「換個 editor」的成本了，它會變成比較像「換平台」的成本。

如果你是公司，這同時是：

- *很爽的槓桿*（交付更快、打斷更少）
- *新的依賴*（你的開發流程跟某個 vendor 綁出形狀）

## 4) 最尷尬但最重要的問題：你買的是速度，還是買了更多不確定性？

我不反對 AI coding 工具。我自己也用。

但我一直看到同一個失控模式：

- 產出變快
- 理解沒有跟著變快

最可怕的不是「AI 寫出 bug」。最可怕的是：**系統變成沒人說得清楚**，然後那個不清楚會變成你永遠的稅。

所以如果你是付錢的人，我反而會要求一些很無聊的東西：

- code review 要更嚴格
- 架構邊界要更清楚
- 測試要真的能抓錯
- 什麼能生成、什麼不能生成，要有政策

不然這類工具就會變成一種乘數器：同時放大生產力，也放大熵。

## 5) 我接下來會看什麼（如果你在意這市場會怎麼走）

如果我把這件事當生意在看（可惜我現在多少得看一點），我會盯這幾個點：

- **定價壓力**：是往低價 per-seat 打，還是往 usage/value-based 走
- **企業化能力**：SSO、資料保留策略、VPC/on-prem、合規 checkbox
- **評估指標**：不要只看「省了多少 token」，要看「cycle time、事故率、返工率」
- **預算到底誰握著**：工程部門？IT？還是財務？

因為最後會形塑產品的，往往不是最會寫 code 的人，而是握著預算的人。

如果 Cursor 真的已經到 20 億 ARR，那這類工具接下來大概率會像雲端早期一樣：魔法感變少，合約與流程變多。
