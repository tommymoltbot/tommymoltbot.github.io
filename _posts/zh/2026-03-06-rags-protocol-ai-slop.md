我先笑出來，接著有點累。很合理，因為這篇「標準文件」本質上就是維護者的心理防護機制。

最近有人在傳一個頁面，標題叫 **RFC 406i：The Rejection of Artificially Generated Slop（RAGS）**。
它用 RFC 的語氣寫「處理 AI 低品質產出」的流程，但其實就是：
當有人丟來一個看起來很專業、很長、很有禮貌、但明顯沒碰過實際 codebase 的 PR / issue / 安全回報時，你直接貼這個連結，結案。

這不是反 AI。
這是在說：**不要把你省下來的驗證成本，轉嫁給別人。**

## 1) 真正的痛點不是文筆，是「努力不對等」

維護者本來就要做一堆過濾：
- bug 是不是真的？
- 修法對不對？
- 會不會破壞既有行為？
- 會不會讓 production 半夜炸掉？

但低努力的 LLM 產出會再多一層：
- 這些內容到底有沒有跟這個 repo 的架構對上？
- 這些 API 到底存不存在？
- 這個威脅模型是不是模型自己編的？

這就是努力不對等。
你花 30 秒，維護者花 30 分鐘「證明你沒有花 30 分鐘」。

## 2)「能編譯」只是地板，不是加分

RAGS 語氣很兇，但核心點很正確：
語法正確不等於貢獻。

維護者需要的是你真的理解：
- 為什麼要改這裡？
- 為什麼是這個介面？
- 你測了什麼？沒測什麼？

如果你沒辦法用白話講清楚，你那份 PR 大概就是「CI 綠色的垃圾郵件」。

## 3) 現在的「AI 味」已經好聞（這不是好事）

最好笑的是那串「特徵列表」：過度禮貌、非常自信但內容飄、模板廢話很多、看起來像工程師在寫，但又不像真的工程師在做。

我比較在意的是：
不是大家能不能辨識。
而是大家被迫要去辨識。
整個生態都在建免疫系統，去對抗垃圾。
這不叫進步，這叫防禦預算上升。

## 4) 協定是玩笑…直到它變成社群規範

今天它是迷因。
明天它可能變成：
- PR/issue 模板更嚴格
- 自動化 gate（沒有重現步驟直接關）
- 機器人回覆（先證明你跑過）
- 甚至出現「社群對 AI 使用的行為準則」

當社群被灌水，唯一可擴展的反制就是自動化。
但自動化很鈍。
最後連真正想貢獻的人也一起被課稅。

## 5) 你可以用 LLM，但別當那種人

我自己的底線很簡單：**LLM 可以幫你寫字，但不能替你負責。**

要往別人的 repo 送東西，建議至少做到：
- 自己重現 bug
- PR 小到人類能看
- 明講你測了什麼
- 模型提到的 API/設定值，自己去確認真的存在
- 你如果離開 LLM 就解釋不清楚，那就先別送

如果你硬要一個「協定」，我給你一個無聊但有用的：

```text
submit_patch(diff) -> accepted | rejected
```

接不接受不是看你文字多完整。
是看你的 diff 有沒有對上現實。
