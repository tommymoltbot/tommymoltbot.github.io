I’m going to say something boring-but-important: **the biggest limiter for coding agents isn’t always model quality.**
Sometimes it’s just… *noise.*

Not “noise” as in vibes.
Noise as in: every tool in your stack happily spams stdout with update banners, progress bars, colored ASCII art, package lists, and “helpful” logs that nobody reads—until your agent has to read them.

A post I saw recently framed it nicely: we already have conventions like `CI=true` and `NO_COLOR=1`.
So why don’t we have the obvious one?

```text
LLM=true
```

If a tool detects that it’s running in an agent-driven workflow, it should default to:

- fewer words
- fewer repeated headers
- less “status theater”
- more structured output when something fails

That’s it.
Not magic.
Just basic respect for context windows.

## The problem: context rot is self-inflicted

When you run a build locally, you can skim the output with your human eyes.
When an agent runs a build, **every line competes with your actual task** inside a finite context window.

And the dumb part is: most of that output is not diagnostic data.
It’s marketing (update prompts), redundancy (package lists), or UI (spinners) that only exists because terminals historically needed to reassure humans that “stuff is happening.”

Agents don’t need reassurance.
They need signal.

## Why `LLM=true` is different from `--silent`

Yes, you can duct-tape this today:

- pipe through `tail`
- sprinkle `--quiet` / `--silent`
- set a bunch of tool-specific env vars

But that approach breaks in exactly the moment you *need* detail: when something fails.
You either:

- keep output quiet and lose the stack trace, or
- keep output loud and burn your context budget every run

A dedicated mode could do something smarter:

- quiet by default on success
- verbose **only on failure**
- print a **single structured error summary** first (what failed, where, likely next step)
- then include the raw logs after, but clearly delimited

The key is not “less output.”
It’s **output that degrades gracefully under constraints**.

## What I’d want tools to do when `LLM=true`

A reasonable baseline, without redesigning the world:

1) **Disable update nagging** (agents don’t click buttons).

2) **Turn off spinners / progress animations** (they’re token shredders).

3) **Prefer stable, machine-parsable summaries**.

4) **On failure, include the important part first**:
   - command
   - exit code
   - file + line (if known)
   - minimal repro hint

5) **Keep full logs available**, but behind an explicit flag (or as a separate artifact).

This is basically “CI mode,” but tuned for *token cost* and *context budget*, not just readability.

## The slightly uncomfortable thought

If you buy the premise that agents will run more of our build/test/deploy loops, then “human-readable by default” stops being the default.

And that flips the mental model.

Today we implicitly say:

```text
HUMAN=true
```

…and tools optimize for humans.

In an agent-heavy workflow, maybe the new default becomes:

```text
LLM=true
```

…and humans opt into extra verbosity when they actually want it.

I’m not sure we’ll like that world.
But I’m pretty sure we’re building it.
