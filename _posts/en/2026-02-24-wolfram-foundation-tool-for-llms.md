LLMs are good at sounding like they know what they’re doing.

They’re *not* good at being precise on demand.

That gap (between “fluent” and “correct”) is why tool use keeps coming back as the boring, durable path forward. Stephen Wolfram just published a long post arguing that foundation models need a **foundation tool**: something broad enough to pair with an LLM’s breadth, but strict enough to do the parts the LLM will always be bad at: exact computation, exact data, and structured reasoning.

I’m not allergic to grand framing, but what caught my eye wasn’t the philosophy. It was the engineering shape of the release: **an MCP service**, plus an “Agent One API”, plus lower-level components.

If you build agent-y systems for real (not just demo scripts), the thing you end up fighting isn’t “can the model call tools?” — it’s *how you standardize tool calling*, *how you version it*, and *how you debug it when it lies to you at 2 a.m.*

## Why a “general tool” matters (even if you already do RAG)

A lot of teams treat tool use as:

- **RAG**: retrieve documents → paste into the prompt → hope the model doesn’t hallucinate.
- **“Calculator tool”**: bolt on a single-purpose function for math.

Wolfram’s pitch is closer to: “stop wiring dozens of one-off tools; give the model a **broad computational substrate** it can lean on.” That’s… actually reasonable.

If the only thing you expose is a tiny set of bespoke endpoints, you’re forcing the model to do planning + translation + verification in one shot.

A general computational tool changes the workflow:

1. Model turns a fuzzy question into a well-posed query.
2. Tool returns exact output (or an exact failure).
3. Model writes the narrative around it.

That middle step is where correctness comes from.

## CAG vs RAG: “infinite documents” is a spicy way to say “compute it”

Wolfram calls their approach **CAG** (computation-augmented generation). Their framing is basically:

- RAG injects text retrieved from documents.
- CAG injects text generated by computation.

I’d rephrase it in a more production-y way:

- RAG is great when your problem is *missing context*.
- CAG is great when your problem is *missing truth*.

If the model needs a current exchange rate, a verified chemical property, a symbolic integral, a graph traversal, a unit conversion, or “what’s the 99th percentile of this distribution?”, you don’t want “retrieved prose”. You want a system that is allowed to be boring and exact.

## MCP is the actual “platform move”

The phrase “MCP service” is the tell.

In 2023, “LLM plugin” meant “everyone invents their own schema, and then you spend your life writing adapters.” In 2026, if you want tool ecosystems to work, you need a **standard protocol** that lets tools be discovered, called, and audited across different model providers and orchestration stacks.

If Wolfram is serious about being a foundation tool, shipping MCP support is the right move because it turns this from:

- *“use Wolfram if you’re in our blessed integration path”*

into:

- *“if your agent speaks MCP, this is just another tool server.”*

That sounds small until you’ve had to maintain 7 different “tool calling” formats across vendors.

## The part I’m still suspicious about

Wolfram has always had two superpowers:

1. A huge, integrated knowledge/computation base.
2. A language that can represent *the shape of computation* (not just inputs/outputs).

The second one is what they’re leaning on when they say the Wolfram Language is a good medium for AIs to “think” computationally.

My suspicion is simpler: this only works if you treat the tool boundary as *a contract*.

The easiest failure mode for agents is:

- Model “calls” a tool but subtly changes the question.
- Tool returns something correct… for the wrong problem.
- Model wraps it in confident prose.

So if you integrate something like this, you need to build around the contract:

```text
tool_call(name, input) -> { output | error }
```

Not fancy. But strict. Log it. Version it. Test it.

## If I were wiring this into a real system

A few practical rules I’d adopt immediately:

- **Force determinism where it matters**: the same input should produce the same tool call, or at least the same *semantic* call.
- **Treat tool errors as first-class**: a clean failure beats a “creative” success.
- **Add an evaluation that checks the tool boundary**: not just “did the final answer look good”, but “did the agent call the right tool with the right parameters.”

The optimistic take is: Wolfram is building a clean “truth layer” under LLMs.

The cynical take is: it’s still an LLM on top, so the last mile is still your problem.

Both can be true.
