![Probability vs Truth](/img/posts/2026-03-05-llm-lying-01.webp)

I saw a post titled **“The L in ‘LLM’ stands for lying”** and my first reaction was: *yeah, that’s dramatic.*

Then I remembered the last time an LLM confidently handed me a “fix” that looked clean, compiled, and was **quietly wrong in a way only production can punish**.

So no, I don’t think models *lie* the way humans lie.

But I *do* think treating LLM output as “statements” is how teams get hurt. Because what you’re really getting is this:

```text
next_token(context) -> token
```

It’s fluent. It’s often useful. And it is **not optimized for being faithful to reality**.

## Thought #1: probability is not truth (even when it reads like it)

A model is rewarded for producing text that *looks like the kind of text people accept*.

That’s an uncomfortable objective if your use-case is:

- incident response
- data migration
- security policy
- “please don’t delete the wrong rows”

When the objective is “be a plausible continuation”, **confident nonsense isn’t a bug**. It’s a predictable failure mode.

## Thought #2: “sounds right” is the most expensive kind of wrong

The cheap wrong answers are the ones that look obviously broken.

The expensive wrong answers are the ones that:

- match the local style of your codebase
- use the right naming conventions
- include comments that sound thoughtful
- pass shallow tests

…and still violate the real constraints (latency, idempotency, concurrency, data invariants, threat model).

This is why vibe coding scares maintainers: it creates a lot of code that *looks* reviewed.

## Thought #3: in production, you don’t need smarter text — you need verifiability

If you want to ship LLMs safely, the design target isn’t “less hallucination”.

It’s:

- **make it cheap to verify**
- **make it hard to do damage**

That means boring engineering:

- run outputs through linters/formatters/typecheckers
- require tests and deterministic repro steps
- add read-only / dry-run modes
- use “plan → apply” flows (and log the plan)

If an LLM can’t explain how to verify a change, treat it like an intern who’s good at writing but hasn’t been burned yet.

## Thought #4: RAG helps, but it doesn’t fix the core incentive

Retrieval grounding is useful. It reduces “invented details”.

But it doesn’t magically turn autocomplete into a truth engine.

You can still get:

- correct citations + wrong reasoning
- correct facts + wrong conclusion
- relevant docs + misread constraints

RAG is a seatbelt. Not a pilot.

## Thought #5: the real shift is cultural — you need a different review muscle

The weird part of LLM-assisted work is that you’re no longer reviewing *someone’s reasoning*.

You’re reviewing an artifact that might have **zero internal coherence**, but still looks like the “right” shape.

So the review habit needs to change:

- ask “what invariant does this preserve?”
- ask “what would make this fail silently?”
- ask “what’s the rollback story?”

That’s not anti-AI. That’s just… production.

## My take

If you want a one-line rule:

> Treat LLM output as *draft code* and *draft claims* — never as authority.

Models are great at generating candidate solutions.

Your job is to build systems where **candidates are cheap and verification is mandatory**.

That’s the only version of “AI productivity” I trust.
