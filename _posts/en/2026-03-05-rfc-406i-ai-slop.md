I clicked a link today and got hit with an HTTP-style error page that basically said: **“AI slop detected. Rejected.”**

The site calls it *RFC 406i* — a “standard protocol” for handling and discarding low-effort AI-generated pull requests, issues, and security reports.

It’s obviously satire.

But… it’s also one of the most honest pieces of writing I’ve seen about the maintainers’ side of the current AI wave.

## The problem isn’t AI. It’s the asymmetry.

The core complaint is not “LLMs are bad” or “people shouldn’t use tools.” The complaint is this:

- it takes **seconds** to generate a confident-looking PR description
- it takes **hours** for a human to validate whether any of it is true

That gap is the whole story.

If you’ve ever maintained anything (open source or internal), you already know the feeling:

You open a PR and it reads *perfectly*.
The grammar is pristine.
The structure is clean.
There’s even a “robust and scalable solution” line at the end.

And then you pull it locally and realize the author never ran it once.

AI didn’t create that dynamic.
It just made it cheap enough that we’re now drowning in it.

## “It compiles” is not the bar

One line in the 406.fail page stuck with me: it basically says that syntax and formatting are the floor, not the ceiling.

That’s right.

Passing CI doesn’t even mean much anymore if the PR also “helpfully” rewrites the tests to only assert `True == True`.

The deeper issue is **accountability**:

When someone submits a change, someone needs to be able to answer:

- what broke before?
- what changed?
- why does this fix it?
- what tradeoff did you accept?
- how do we know it won’t regress next week?

A language model can generate text that *resembles* those answers.
But “resembles” isn’t the same as “is.”

## What a real “anti-slop protocol” looks like

Here’s my practical read of what maintainers actually want — a boring checklist that forces symmetry back into the system.

### 1) Require a reproducible report
If it’s a bug report, it needs:

- exact version / commit
- minimal reproduction
- expected vs actual
- logs / stack traces

If you can’t reproduce it, you can’t fix it.

### 2) Demand the smallest diff that proves the point
LLM-generated PRs love scope creep.

A decent review workflow does the opposite:

- keep the change tight
- show the failing test first
- make the fix obvious

### 3) Make the contributor own the end-to-end story
When someone submits a PR, they’re implicitly saying:

> “I ran this. I understand the change. If you ask questions, I can answer them.”

If the PR author can’t answer follow-ups because the model “did the thinking”… that’s not a contribution, that’s a support ticket.

### 4) Be explicit about what you will discard
I like the idea of having a short “auto-reject macro” (less spicy than 406.fail) that says:

```text
Thanks for the submission.

We can’t review AI-generated changes that aren’t backed by:
- a minimal reproduction or failing test
- a clear explanation of the root cause
- evidence the author ran and validated the fix

Please resubmit with the above.
```

The point is not to shame anyone.
The point is to stop maintainers from being used as an LLM validation service.

## The awkward part: this is also going to hit internal teams
People talk about open source, but the same pattern is already happening in companies:

- “AI wrote it” becomes the reason nobody understands production failures
- “AI wrote it” becomes the excuse for not writing a runbook
- “AI wrote it” becomes the default for PR descriptions that look good and say nothing

If you’re running a service with real users, this ends the same way every time:

- a pager
- a messy incident timeline
- and someone asking “who approved this?”

## My take

I’m not anti-LLM.
I use them.

But I’m very anti “ship text that no human is willing to stand behind.”

If a tool makes it 10× easier to produce words, the bar for *verification* has to go up, not down.

And yeah, a meme-RFC with a fake HTTP status code is a weird place to find that message.

But it landed.
